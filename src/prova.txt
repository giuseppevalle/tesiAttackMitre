			<h1 id="firstHeading" class="firstHeading" lang="en">Launch Daemon</h1>
									<div id="bodyContent" class="mw-body-content">
									<div id="siteSub">From enterprise</div>
								<div id="contentSub"></div>
												<div id="jump-to-nav" class="mw-jump">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
				</div>
				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="hf-nsheader"></div><div class="hf-header"></div><table class="infobox" cellspacing="3" style="border-spacing:3px;width:22em;"><tr><th colspan="2" style="text-align:center;font-size:125%;font-weight:bold;">Launch Daemon</th></tr><tr><th colspan="2" style="text-align:center;">Technique</th></tr><tr><th scope="row" style="text-align:left;">ID</th><td>
T1160</td></tr><tr><th scope="row" style="text-align:left;">Tactic</th><td>
Persistence, Privilege Escalation</td></tr><tr><th scope="row" style="text-align:left;">Platform</th><td>
<span class="nowrap">MacOS</span>, <span class="nowrap">OS X</span></td></tr><tr><th scope="row" style="text-align:left;">Permissions Required</th><td>
Administrator</td></tr><tr><th scope="row" style="text-align:left;">Effective Permissions</th><td>
root</td></tr><tr><th scope="row" style="text-align:left;">Data Sources</th><td>
<span class="nowrap">Process Monitoring</span>, <span class="nowrap">File monitoring</span></td></tr></table>
<p>Per Appleâ€™s developer documentation, when macOS and OS X boot up, launchd is run to finish system initialization. This process loads the parameters for each launch-on-demand system-level daemon from the property list (plist) files found in <code>/System/Library/LaunchDaemons</code> and <code>/Library/LaunchDaemons</code><span id="scite-ref-418393c65b829ecd4eef686e5a288625-1-a" class="scite-citeref-number" data-reference="AppleDocs Launch Agent Daemons"><a href="#scite-418393c65b829ecd4eef686e5a288625">1</a></span>. These LaunchDaemons have property list files which point to the executables that will be launched<span id="scite-ref-58a706cc061511d9d15afd2d7f6ba6d6-2-a" class="scite-citeref-number" data-reference="Methods of Mac Malware Persistence"><a href="#scite-58a706cc061511d9d15afd2d7f6ba6d6">2</a></span>.
</p><p>Adversaries may install a new launch daemon that can be configured to execute at startup by using launchd or launchctl to load a plist into the appropriate directories<span id="scite-ref-3e272e4e7bc4ceb79b39657db36a4154-3-a" class="scite-citeref-number" data-reference="OSX Malware Detection"><a href="#scite-3e272e4e7bc4ceb79b39657db36a4154">3</a></span>. The daemon name may be disguised by using a name from a related operating system or benign software <span id="scite-ref-cd67d5a08cb8e6eaaf3ba4731fe2e88e-4-a" class="scite-citeref-number" data-reference="WireLurker"><a href="#scite-cd67d5a08cb8e6eaaf3ba4731fe2e88e">4</a></span>. Launch Daemons may be created with administrator privileges, but are executed under root privileges, so an adversary may also use a service to escalate privileges from administrator to root.
</p><p>The plist file permissions must be root:wheel, but the script or program that it points to has no such requirement. So, it is possible for poor configurations to allow an adversary to modify a current Launch Daemonâ€™s executable and gain persistence or Privilege Escalation.
</p><p>&#32;

</p>
<h2><span class="mw-headline" id="Mitigation">Mitigation</span></h2>
<p>Limit privileges of user accounts and remediate Privilege Escalation vectors so only authorized administrators can create new Launch Daemons.
</p>
<h2><span class="mw-headline" id="Detection">Detection</span></h2>
<p>Monitor Launch Daemon creation through additional plist files and utilities such as Objective-See's Knock Knock application.
</p>
